LangChain, LlamaIndex & Agents

From writing every API call yourself to letting an LLM choose what to do next

For five sections you've built GenAI applications by hand — calling the OpenAI client directly, wiring up your own retry logic, your own ChromaDB calls, your own prompt templates. That was deliberate. You can't reason about a framework's abstractions until you've written the thing the framework is abstracting.

But once you've built RAG from scratch, a pattern becomes obvious: every real GenAI app is built from the same dozen pieces — load documents, chunk them, embed them, search, format a prompt, call the LLM, parse, maybe loop, maybe call a tool, maybe remember. Frameworks exist so you don't rebuild those pieces every project.

🛠️
What this section covers: The two dominant GenAI frameworks (LangChain and LlamaIndex), how to compose them with LCEL, and the centerpiece of modern GenAI engineering — building agents that decide their own steps using tools.

Chapter 1

Why Frameworks Exist

The case for embracing abstraction — once you've earned it

You've built GenAI applications by hand for five sections. Now look at what you wrote: prompt templates assembled with f-strings, a retry loop you copy-pasted between notebooks, ChromaDB calls duplicated across files, provider-specific code that breaks when you swap from OpenAI to Anthropic. Every one of those is a primitive that frameworks already give you.

Building a GenAI app from scratch is like building a web app without a framework. You can handle routing, sessions, templating, and database connections by hand using only the standard library. People did, in 1998. Once Django and Rails existed, nobody chose to keep doing it the hard way — except for very small scripts or very particular performance reasons. LangChain and LlamaIndex are the Django and Rails of GenAI.
Why frameworks now

Three things tip you over from "I'll write it myself" to "I want a framework":

  1. Composition. When your app does six things in sequence, wiring them up by hand becomes the bug surface.
  2. Swappability. Today GPT-4o, tomorrow Claude, next month a local Llama. Frameworks abstract the provider so the rest of your code doesn't change.
  3. Agents. The moment you want an LLM to decide what to do next, you need orchestration that's hard to write well from scratch.

What you get out of the box

  • Prompt management — templates, partial variables, output parsers
  • Model abstraction — one interface across OpenAI, Anthropic, Groq, Ollama
  • Chains — pipelines that compose multiple steps cleanly
  • Memory — conversation state across turns
  • Retrievers — pluggable search over any vector store
  • Agents — LLMs that can choose tools and act in loops
  • Observability — built-in tracing, evaluation, debugging hooks

This section is about embracing those abstractions — knowing what they give you, how to use them, and where each framework shines.

Chapter 2

LangChain vs LlamaIndex — A Mental Model

Two frameworks, two different shapes of problem

People often ask "which one should I learn?" — as if it's a single choice. It's not. They're optimized for different problems and most production apps end up using both.

Think of it like kitchens. LangChain is a fully equipped restaurant kitchen — every appliance, every gadget, every connection. Built for any kind of dish. LlamaIndex is a specialized bakery — laser-focused on one craft (data → retrieval → answers) and exceptionally good at it. You can bake bread in either, but the bakery has the better oven.

Origin shapes the framework

LangChain started as "let's compose LLM calls into chains" and grew into a general-purpose orchestration framework. Its sweet spot is anything involving reasoning, decisions, or external tools — agents, multi-step workflows, complex pipelines that aren't just RAG.

LlamaIndex started as "let's connect LLMs to your data" (originally called GPT Index). Its sweet spot is RAG and retrieval — ingesting weird data sources, building sophisticated indices, and answering questions over them. Its retrieval primitives are deeper.

DimensionLangChainLlamaIndex
Core strengthOrchestration, agents, chainsData ingestion, indexing, retrieval
Mental modelComposable Lego blocks (LCEL)Data → Index → Query Engine
Data connectors~80 loaders300+ via LlamaHub
Index typesVector (mostly)Vector, Tree, Keyword, Graph, Hybrid
AgentsMature, many patternsAvailable, less central
TracingLangSmith (first-party)Multiple integrations
Best forAgentic apps, tool use, complex flowsProduction RAG over messy data
The honest answer

You'll likely use both in real projects. LlamaIndex for the retrieval layer, LangChain for the orchestration around it. They're explicitly interoperable — LlamaIndex query engines can be wrapped as LangChain tools, and vice versa.

Chapter 3

LCEL — The Pipe Operator That Changed LangChain

Components compose like Unix commands

If you read LangChain code from 2023 and 2024, it looks like two different libraries. That's because of LCEL — LangChain Expression Language. It's not a new language; it's a way of composing components using Python's | operator, the same way you pipe commands in a shell.

In Unix: cat file.txt | grep error | wc -l. Each command does one thing, output flows left to right. LCEL is exactly that, but for LLM building blocks. prompt | model | parser reads top to bottom: take input, format it as a prompt, send to the model, parse the output.

Before LCEL (the old way)

from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

prompt = PromptTemplate.from_template("Translate '{text}' to French")
chain = LLMChain(llm=llm, prompt=prompt)
result = chain.run(text="Hello, world")

With LCEL (the modern way)

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("Translate '{text}' to French")
chain = prompt | llm | StrOutputParser()
result = chain.invoke({"text": "Hello, world"})
What LCEL gives you for free
  • Streaming — chain.stream(...) works automatically
  • Async — chain.ainvoke(...) works automatically
  • Batching — chain.batch([...]) works automatically
  • Tracing — every step shows up in LangSmith
  • Composability — chains pipe into other chains
Interactive · LCELPipeDemo

From here on, when you see LangChain code in this section, it'll be in LCEL. It's the only style worth learning — the older imperative chains are deprecated.

Chapter 4

Chains in Practice

Sequential, parallel, and routing patterns

A "chain" is just a pipeline. The interesting question is what patterns of chaining come up over and over.

Pattern 1: Sequential refinement

Use the output of one LLM call as the input to the next. Classic example: generate a draft, critique it, revise.

draft_chain = draft_prompt | llm | StrOutputParser()
critique_chain = critique_prompt | llm | StrOutputParser()
revise_chain = revise_prompt | llm | StrOutputParser()

full = (
    {"topic": RunnablePassthrough()}
    | {"draft": draft_chain, "topic": lambda x: x["topic"]}
    | {"critique": critique_chain, "draft": lambda x: x["draft"]}
    | revise_chain
)

Pattern 2: Parallel fan-out

Run multiple chains at once and combine results. Useful for asking the same question from different angles.

from langchain_core.runnables import RunnableParallel

multi_view = RunnableParallel(
    pros=pros_chain,
    cons=cons_chain,
    summary=summary_chain
)
result = multi_view.invoke({"topic": "remote work"})
# result = {"pros": "...", "cons": "...", "summary": "..."}

Pattern 3: Routing

Pick a different chain based on the input. The LLM acts as a classifier, then a specialized chain handles each case.

def route(info):
    if "code" in info["topic"].lower():
        return code_chain
    elif "math" in info["topic"].lower():
        return math_chain
    return general_chain

router = (
    {"topic": classifier_chain, "input": lambda x: x["question"]}
    | RunnableLambda(route)
)
When to reach for a chain vs. plain Python

If your "chain" is two LLM calls with simple Python in between, just write Python. Use chain abstractions when you want streaming, async, batching, or tracing for free across the whole pipeline — that's where LCEL pays for itself.

Chapter 5

Memory

Giving your chain a past

An LLM call is stateless. The model doesn't remember the previous turn. If you want a chatbot, you have to hand the model the conversation history every time. Memory is the abstraction for managing that history.

Memory is a notebook the assistant carries into every meeting. Before answering, the assistant flips it open and reads what was discussed before. After answering, the assistant writes down what just happened. The model itself never "remembers" — the notebook does.

The four memory types you'll actually use

Interactive · MemoryTypesDemo
TypeWhat it storesWhen to use
BufferFull conversation, verbatimShort chats, when you have token budget
WindowLast N turns onlyLong chats where only recent matters
SummaryRunning LLM-generated summaryVery long chats, lossy is OK
Vector-backedEmbedded turns, retrieved on relevanceChats where old context might matter again

The modern way: RunnableWithMessageHistory

from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_core.chat_history import InMemoryChatMessageHistory

store = {}
def get_history(session_id: str):
    if session_id not in store:
        store[session_id] = InMemoryChatMessageHistory()
    return store[session_id]

chain_with_memory = RunnableWithMessageHistory(
    chain,
    get_history,
    input_messages_key="input",
    history_messages_key="history",
)

chain_with_memory.invoke(
    {"input": "What's the capital of France?"},
    config={"configurable": {"session_id": "user-42"}}
)
The thing nobody tells you

In production, you almost never use InMemoryChatMessageHistory. You back it with Redis, Postgres, or DynamoDB so chats survive restarts and scale across servers. LangChain has integrations for all of these — same interface, different storage.

Chapter 6

Tools

Giving the LLM hands

An LLM by itself can only produce text. A tool is any function you let the LLM call — search the web, look up a stock price, query your database, send an email, run code. Tools are how an LLM does things rather than just talking about them.

An LLM without tools is a brilliant consultant locked in a room with no phone. It can advise, reason, write — but it can't verify a current fact, can't check your calendar, can't do anything outside its own training data. Tools are the phone, the browser, the database access.

Defining a tool in LangChain

from langchain_core.tools import tool

@tool
def get_stock_price(ticker: str) -> str:
    """Get the current stock price for a ticker symbol like AAPL or TSLA."""
    # In real life: call a market data API
    prices = {"AAPL": 224.50, "TSLA": 248.10, "GOOGL": 178.30}
    if ticker in prices:
        return f"{ticker} is trading at ${prices[ticker]}"
    return f"No price found for {ticker}"

That's it. The @tool decorator turns a Python function into something an LLM can call. The docstring becomes the tool's description — this is what the LLM reads to decide whether to use it, so write it carefully.

Binding tools to a model

from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")
llm_with_tools = llm.bind_tools([get_stock_price])

response = llm_with_tools.invoke("What's Apple trading at right now?")
# response.tool_calls = [{'name': 'get_stock_price', 'args': {'ticker': 'AAPL'}, ...}]

The model doesn't execute the tool. It produces a structured request: "I want to call get_stock_price with ticker='AAPL'." Your code is responsible for actually running the function and feeding the result back.

The tool-calling loop
  1. User sends a message
  2. LLM responds with either a final answer or a tool call
  3. If tool call: your code runs the tool, sends the result back
  4. LLM sees the result, responds again — final answer or another tool call
  5. Repeat until the LLM gives a final answer

This loop is what an agent is. We'll build one in Chapter 13.

Tools you don't have to write

LangChain ships with hundreds of pre-built tools: web search (Tavily, DuckDuckGo, Google Serper), code execution (Python REPL), Wikipedia, Wolfram Alpha, SQL databases, file systems, shell commands. Most production agents use a mix of pre-built and custom tools.

Chapter 7

Callbacks & LangSmith

Seeing inside the box

The single hardest thing about debugging GenAI apps is that everything is a black box by default. Your chain runs, something comes out, and if it's wrong you have no idea which step broke. Callbacks and tracing fix that.

Without tracing, debugging a chain is like debugging a program with no print statements. Tracing gives you a complete X-ray of every step: prompts in, tokens out, latency, cost, tool calls, retries — all of it, automatically.

LangSmith — the production observability tool

LangSmith is LangChain's hosted tracing platform. You add two environment variables and every chain run is logged with full inputs, outputs, intermediate steps, latency, and cost.

import os
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"] = "ls__..."
os.environ["LANGCHAIN_PROJECT"] = "my-rag-app"

# That's it. Every chain.invoke() now appears in LangSmith.

What you actually use it for

  • Debugging — when output is wrong, see exactly which prompt caused it
  • Latency analysis — find which step is slow
  • Cost tracking — see token usage per chain, per user, per endpoint
  • Eval datasets — promote good/bad runs into test cases
  • A/B testing — compare two prompt versions on the same inputs
Production rule

If your GenAI app is going anywhere near a real user, turn on tracing from day one. The cost of not having traces when something goes wrong in production is far higher than the cost of running them.

Chapter 8

Data Connectors

LlamaIndex's killer feature

If LangChain's superpower is orchestration, LlamaIndex's superpower is getting your data in. The LlamaHub registry has 300+ connectors for almost any data source you can name.

Building a RAG system, the messy part is rarely the embeddings or the vector store. It's getting Slack messages, Notion pages, PDFs, Confluence wikis, Salesforce records, and Postgres rows into a uniform format. LlamaIndex connectors are like having a USB-C adapter for every weird port.

The interface is uniform

Every connector returns a list of Document objects. Once you have documents, the rest of the pipeline is the same — regardless of where the data came from.

from llama_index.core import SimpleDirectoryReader
from llama_index.readers.notion import NotionPageReader
from llama_index.readers.web import SimpleWebPageReader

# Local files
docs = SimpleDirectoryReader("./data").load_data()

# Notion
docs = NotionPageReader(integration_token=TOKEN).load_data(page_ids=[...])

# Web pages
docs = SimpleWebPageReader().load_data(["https://example.com/article"])

# All return List[Document] — same shape, different sources

Connectors that solve real problems

  • SimpleDirectoryReader — handles PDF, DOCX, MD, TXT, CSV, images automatically
  • SECFilingsReader — pulls company 10-K, 10-Q filings parsed into sections
  • SlackReader, NotionReader, ConfluenceReader — the obvious enterprise sources
  • DatabaseReader — runs SQL, returns rows as documents
  • UnstructuredReader — wraps Unstructured.io for messy real-world docs

This is where LlamaIndex genuinely beats LangChain. If your project starts with "we have data in 5 different systems and need to RAG over it," LlamaIndex is the right starting point.

Chapter 9

Indices

More than just vector search

In Sections 4 and 5, "the index" meant one thing: a vector store. LlamaIndex generalizes the idea. An index is any data structure built over your documents to make retrieval efficient — and there are several worth knowing.

Interactive · IndexComparisonDemo

The four index types you'll meet

VectorStoreIndex — the default. Embed each chunk, store in a vector DB, retrieve by semantic similarity. Exactly what you built in Section 4.

SummaryIndex (formerly ListIndex) — stores all chunks in order. At query time, the LLM reads through them sequentially. Useful when the answer might be anywhere and you want exhaustive coverage. Slower, but no retrieval miss possible.

TreeIndex — builds a hierarchical summary tree over chunks. The LLM traverses top-down, drilling into the relevant branch. Useful for very large document sets where summaries-of-summaries help.

KeywordTableIndex — builds an inverted index of keywords per chunk. Pure lexical retrieval, no embeddings. Useful as a hybrid component or when semantic search misses exact-match queries (product codes, IDs, error codes).

The simplest possible LlamaIndex RAG

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

documents = SimpleDirectoryReader("./data").load_data()
index = VectorStoreIndex.from_documents(documents)

query_engine = index.as_query_engine()
response = query_engine.query("What's our return policy?")
print(response)

Three lines from raw files to a working RAG system. LlamaIndex has opinions baked in — sensible chunking, default embedding model, default retrieval — so you can start fast and customize as needed.

When to override defaults

Start with defaults. The moment retrieval quality matters (and it always does), customize the chunker, embedding model, retrieval count (similarity_top_k), and response synthesizer. The whole pipeline is configurable — the defaults just let you skip configuration when prototyping.

Chapter 10

Query Engines & Routers

Multiple ways to ask the same index

An index stores data. A query engine is what you query. The same index can be queried in multiple ways — different retrieval strategies, different response synthesis modes, different post-processing.

Query engine modes

# Default: retrieve top-k, stuff into one prompt, generate
qe = index.as_query_engine()

# Tree summarize: retrieve, summarize in chunks, then summarize summaries
qe = index.as_query_engine(response_mode="tree_summarize")

# Refine: iteratively refine an answer chunk by chunk
qe = index.as_query_engine(response_mode="refine")

# With re-ranking
from llama_index.core.postprocessor import SentenceTransformerRerank
qe = index.as_query_engine(
    similarity_top_k=10,
    node_postprocessors=[SentenceTransformerRerank(top_n=3)]
)

Routers — the LlamaIndex answer to "which index?"

Real apps have multiple data sources — product docs in one index, support tickets in another, code in a third. A router query engine uses an LLM to pick the right one.

from llama_index.core.tools import QueryEngineTool
from llama_index.core.query_engine import RouterQueryEngine

product_tool = QueryEngineTool.from_defaults(
    query_engine=product_qe,
    description="Use for questions about product features and pricing"
)
support_tool = QueryEngineTool.from_defaults(
    query_engine=support_qe,
    description="Use for questions about troubleshooting and known issues"
)

router = RouterQueryEngine.from_defaults(
    query_engine_tools=[product_tool, support_tool]
)

router.query("My device won't turn on")  # routes to support_tool
The router is just an agent in disguise

A router is an LLM picking between options based on descriptions — that's the same primitive we'll use for full agents in the next group. Routers are agents with one decision; agents are routers with many.

Chapter 11

What Is an Agent?

An LLM in a loop, with tools, choosing what to do next

Until now, the LLM has been a function: input goes in, output comes out, you decide what happens next. An agent flips that. The LLM decides what happens next.

A regular chain is a recipe — fixed steps, in order, every time. An agent is a chef. You hand it a goal ("make me dinner") and a kitchen full of tools (knife, stove, ingredients), and it figures out the steps as it goes — checking the fridge, deciding to pivot, tasting and adjusting. The control flow is no longer in your code; it's in the model's head.

The minimum viable definition

An agent is an LLM in a loop with three things:

  1. A goal — the user's request
  2. A set of tools — functions it can call
  3. A loop — it can call tools, see results, and decide what to do next
Interactive · AgentLoopDemo

Why this is a big deal

With chains, you wrote the orchestration. "First do X, then Y, then Z." With agents, the LLM writes the orchestration at runtime. The same agent can answer "what's Apple's stock price?" (one tool call) or "compare Apple's stock to its 5-year average and explain what changed" (many tool calls, including math) without you writing different code for each.

When agents are the right answer

Use an agent when:

  • The number of steps depends on the input — you don't know it ahead of time
  • The tools needed depend on the input — you don't know which subset
  • The order of tools depends on results — branching logic that's hard to write by hand

If your flow is "always call A then B then C," it's a chain, not an agent. Don't over-engineer.

Chapter 12

The ReAct Loop

How agents actually think

The dominant agent pattern is called ReAct (Reasoning + Acting), introduced in a 2022 paper. It's the loop nearly every modern agent uses, even when the framework hides it from you.

The pattern

At each step, the LLM produces three things:

  • Thought — what it's reasoning about right now
  • Action — which tool to call, with what arguments (or "I'm done, here's the answer")
  • Observation — the result that comes back from the tool

Then the cycle repeats: the next Thought now has access to the previous Observation.

What a real ReAct trace looks like

Interactive · ReActTraceDemo

The two failure modes

Infinite loops. The agent keeps calling tools, never decides it's done. Fix: set max_iterations. Always.

Wrong tool choice. The agent picks the wrong tool because the descriptions are ambiguous. Fix: write tool descriptions like you're writing a job posting — specific, concrete, with examples of when to use it.

Tool description rule of thumb

Bad: "Searches the web"

Good: "Searches the web for current events, news, prices, sports scores, or any information that may have changed after January 2024. Do NOT use for general knowledge or math."

The model reads these descriptions to decide. Treat them like prompts within prompts.

Chapter 13

Building a Real Agent

Three tools, one loop, all the pieces together

Time to build the thing. We'll create an agent with three tools — web search, a calculator, and a custom API call — then watch it reason through a question that needs all three.

The setup

from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from tavily import TavilyClient
import requests

# Tool 1: Web search
tavily = TavilyClient(api_key=TAVILY_KEY)

@tool
def web_search(query: str) -> str:
    """Search the web for current events, news, prices, or recent information.
    Returns a summary of top results."""
    results = tavily.search(query, max_results=3)
    return "\n".join([r["content"] for r in results["results"]])

# Tool 2: Calculator
@tool
def calculate(expression: str) -> str:
    """Evaluate a mathematical expression. Use Python syntax.
    Example: calculate('(224.50 - 180.30) / 180.30 * 100')"""
    try:
        return str(eval(expression, {"__builtins__": {}}, {}))
    except Exception as e:
        return f"Error: {e}"

# Tool 3: Custom API — exchange rates
@tool
def get_exchange_rate(from_currency: str, to_currency: str) -> str:
    """Get current exchange rate between two currencies (3-letter codes like USD, EUR, AED)."""
    r = requests.get(f"https://api.exchangerate-api.com/v4/latest/{from_currency}")
    rate = r.json()["rates"].get(to_currency)
    return f"1 {from_currency} = {rate} {to_currency}" if rate else "Currency not found"

Wiring it up

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
tools = [web_search, calculate, get_exchange_rate]

agent = create_react_agent(llm, tools)

result = agent.invoke({
    "messages": [("user",
        "What's Apple's current stock price in AED, "
        "and how much higher is it than $200?")]
})

for msg in result["messages"]:
    print(msg.content)

What happens inside

The agent sees the question and reasons (paraphrased):

  • "I need Apple's stock price — that's current data, use web_search"
  • Calls web_search("Apple stock price today") → gets ~$224.50
  • "I need this in AED — use get_exchange_rate"
  • Calls get_exchange_rate("USD", "AED") → gets 3.67
  • "Now the math — use calculate"
  • Calls calculate("224.50 * 3.67") → 824.0
  • Calls calculate("224.50 - 200") → 24.50
  • Returns: "AAPL is at $224.50, or about 824 AED, which is $24.50 above $200."

You wrote three small functions and one create_react_agent call. The orchestration — picking tools, ordering calls, threading results — is all the LLM. That's the leverage frameworks give you.

LangGraph note

LangChain's modern agent runtime is LangGraph — a graph-based orchestration layer that replaced the older AgentExecutor. It's more flexible (custom state machines, human-in-the-loop checkpoints, parallel tool calls). For most cases, create_react_agent is the one-liner you want. Drop down to raw graphs when you need explicit control flow.

The framework fatigue conversation

You'll hear engineers say "frameworks are leaky abstractions, just write raw API calls." There's a kernel of truth — frameworks evolve fast, breaking changes happen, and for very simple apps the framework is more code than the code. But this view ignores three things:

  • Agents are genuinely hard to build well from scratch. Tool calling, state management, retry logic, parallel execution, streaming — by the time you've built it yourself, you've reinvented LangGraph.
  • Observability is huge. LangSmith alone justifies the framework for most teams.
  • Provider abstraction matters. When OpenAI raises prices or Anthropic ships a better model, swapping is one line.
My take

Use frameworks. Embrace them. Read the source code when you hit a leaky abstraction (it happens), and don't be afraid to drop down to raw APIs for performance-critical paths. But treating "no frameworks" as a virtue costs you weeks of building primitives that are already built.

You earned the right to use frameworks by writing the raw version first. Now go faster.
